# Learnings

- **2026-07-05** — Live verification (2026-07-05): the duckdns frontend, Vercel frontend, Render `/health`, `/api/search`, `/api/speech/transcribe`, and `/api/analytics/events` all returned 200 from an external probe — but the **ingest path 502s because `WEBSHARE_PROXY_URL` is not set on Render** (only in local `.env`). The proxy fix from b1e3a3d is deployed, but the env var itself never made it to Render. The transcript endpoint's error message names `WEBSHARE_PROXY_USERNAME`/`WEBSHARE_PROXY_PASSWORD`; all three forms are valid (`*_URL` takes priority). Without one of them set on Render, live ingestion is guaranteed to fail — set it before any demo. Captured in `DEMO_DAY_RUNBOOK.md` step 0.
- **2026-07-05** — After a DNS cutover, local tests lie: this PC's resolver chain (router/VPN ignoring DuckDNS's 60s TTL) kept serving the dead EC2 IP for over an hour, first faking success (old server still up) and then faking failure (timeout after teardown). Verify cutover with EXTERNAL probes: `check-host.net/check-http` (multi-country HTTP checks) and `crt.sh` (proves cert issuance). Local `ipconfig /flushdns` doesn't help when the staleness is upstream; browser "Secure DNS" (DoH) bypasses it.
- **2026-07-05** — `curl.exe --resolve <host>:443:<ip>` TLS probes fail with schannel exit 35 on this machine even against known-good endpoints — likely Avast HTTPS scanning interfering. Don't trust local direct-IP TLS probes here; they falsely suggested Vercel had no certificate.
- **2026-07-05** — When adding an apex-style domain on Vercel (incl. PSL domains like `*.duckdns.org`), UNCHECK "Redirect apex domains to www" — DuckDNS cannot serve the `www.` variant, and the checkbox silently creates a broken redirect target.

- **2026-07-05** — pydantic-settings JSON-decodes `list[str]` fields from env vars BEFORE field validators run, so a `mode="before"` validator cannot rescue non-JSON values. `CORS_ORIGINS=https://...` (plain string) crashed startup on Render with a SettingsError until the field was annotated with `NoDecode` (fixed in 5b64f54). Symptom in browsers: unhandled 500s bypass CORSMiddleware, so fetch fails with an opaque "Failed to fetch" instead of a readable error.
- **2026-07-05** — YouTube blocks transcript requests from Render's datacenter IPs just like EC2's; `youtube-transcript-api` raises `RequestBlocked`/`IpBlocked` (subclasses of `CouldNotRetrieveTranscript`), which the service didn't catch → instant 500 on ingest. Fixed in b1e3a3d (clean 502 with proxy hint). The working fix is `WEBSHARE_PROXY_URL` in the environment.
- **2026-07-05** — DuckDNS + Vercel works: `duckdns.org` is on the Public Suffix List, so Vercel treats `asktube-ai.duckdns.org` as an apex domain and accepts an A record (216.198.79.1) — DuckDNS can't do CNAMEs. Cert issued within minutes.
- **2026-07-05** — 9 speech-route tests fail with 503 unless `OPENAI_API_KEY` is set (any dummy value works) — the route checks global settings at request time. Fresh test environments must export it before `pytest`.
- **2026-07-05** — Render deploys the `backend/` root dir with the Docker runtime (ffmpeg included, port 8000 via `PORT` env). The service URL got a suffix: `asktube-ai-q2gi.onrender.com` — a wrong hostname baked into Vercel's `NEXT_PUBLIC_API_URL` fails silently ("Search paused" UI error); the runtime fallback in api.ts only rescues *localhost* URLs, not wrong public ones.

- **2026-07-04** — `frontend/lib/analytics.ts` must NOT import from `@/lib/api`: the voice test suites (`__tests__/floating-companion-voice.test.tsx`, `__tests__/search-console-voice.test.tsx`) stub the whole module with `vi.mock("@/lib/api", () => ({...}))`, so any extra export pulled in by `analytics.ts` resolves to `undefined` and crashes `trackAnalyticsEvent` inside components. Keep the API-base fallback logic duplicated but in sync between `resolveApiBase` (api.ts) and `resolveAnalyticsApiBase` (analytics.ts).
- **2026-07-04** — In docker-compose, keys under `environment:` override values from `env_file:`. `CORS_ORIGINS` is therefore interpolated (`${CORS_ORIGINS:-...}`) in `docker-compose.yml` so a production `.env` value actually reaches the backend container.
- **2026-07-04** — The live EC2 demo maps the frontend to host port `3001` via a local override that is NOT in the repo; the repo's `docker-compose.yml` maps `3000:3000`. The Nginx snippet in DEPLOY.md proxies to `3001` — deploying from a clean checkout without adjusting one side yields 502 Bad Gateway (note added to DEPLOY.md §"Current live demo setup").
- **2026-07-04** — On Windows, `greenlet` fails to import (`DLL load failed ... filename or extension is too long`) when the venv lives under a very long path. Backend tests run fine from a venv at a short path (e.g. `C:\Users\sisaz\AppData\Local\Temp\at_venv`); only `pydantic`, `pydantic-settings`, `sqlalchemy`, `aiosqlite`, `greenlet`, `pytest`, `pytest-asyncio` are needed for `tests/test_analytics_service.py`.
